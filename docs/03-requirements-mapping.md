## Core requirements

| Requirement (assignment wording) | How it is implemented | Where | Evidence |
|---|---|---|---|
| **Provision the required infrastructure components** — compute, load balancers, databases, storage | Everything is Terraform, in two roots: `bootstrap/` (state bucket, 13 ECR repos, OIDC, budget — applied once) and `terraform/` (VPC, ALB, ECS cluster + 13 Fargate services, RDS MySQL, monitoring, auto-scaling — rebuilt at will). | `bootstrap/` · `terraform/envs/dev/` · `terraform/modules/{network, alb, ecs-cluster, ecs-service, rds, monitoring}` | [plan artifact](img/plan-artifact.png) · [storefront on the ALB URL](img/storefront-alb.png) |
| **High availability** | Two availability zones end to end: ALB nodes in both public subnets; front-end holds a **minimum of 2 tasks** spread across AZs (auto-scaling floor); RDS `multi_az` provides a synchronous standby with automatic failover; ALB health checks gate rotation so only healthy tasks receive traffic. | `terraform/modules/network` (2×2 subnets) · `terraform/envs/dev/autoscaling.tf` (`min_capacity = 2`) · `terraform/modules/rds` (`multi_az`) · `terraform/modules/alb` (health check) | [target group: 2 healthy targets across AZs](img/tg-two-az.png) · [RDS Multi-AZ](img/rds-multiaz.png) |
| **Scalability** | Target-tracking auto-scaling on the front-end (2 → 5 tasks) driven by **two application metrics**: average CPU @ 60% and ALB requests-per-target @ 100. | `terraform/envs/dev/autoscaling.tf` · load generator: `terraform/envs/dev/loadtest.tf` + `load-demo` pipeline job | [dashboard during load: task count stepping 2→5, p99 flat](img/scale-2-to-5.png) |
| **Fault tolerance** | Four independent layers: ECS desired-count reconciliation replaces any dead task automatically; the ALB drains connections (30s) and ejects unhealthy targets; the **deployment circuit breaker halts a bad rollout and auto-rolls back** to the last healthy revision; RDS Multi-AZ fails over the database. Demonstrated live: killing a front-end task mid-traffic — the site never dropped a request while ECS launched the replacement. | `terraform/modules/ecs-service` (circuit breaker, `deregistration_delay`) · `terraform/modules/rds` | [kill-task healing: draining target + replacement entering rotation](img/kill-task-healing.png) |
| **Deploy the application in an automated and repeatable manner** | GitLab CI end to end: images are pulled, scanned, and pushed to ECR by the pipeline; services deployed via `apply` button; the database seeds itself via the `seed` button (fails unless `SEED COMPLETE` appears in the task's own logs). Repeatability is measured, not claimed: the environment can be destroyed and rebuilt with three button presses in **~⟨10 minutes⟩** — images and state survive in bootstrap, so rebuilds never repeat the mirror. | `.gitlab-ci.yml` · seeding: `terraform/envs/dev/seed.tf` + `seed/dump.sql` | [green pipeline](img/pipeline-green.png) · [SEED COMPLETE in the job log](img/seed-complete.png) |
| **Monitoring and logging solutions** | Every container's stdout/stderr ships to per-service CloudWatch log groups (7-day retention, Terraform-managed); Container Insights adds per-service/task metrics; a 7-widget golden-signals dashboard charts **p50 and p99 on the same axis** plus errors, traffic, saturation, and RDS vitals; **five alarms** (with recovery notifications) deliver by email — including a latency alarm that tripped and recovered organically during load testing. | `terraform/modules/monitoring` · `terraform/modules/ecs-service` (log groups, awslogs) · `terraform/modules/ecs-cluster` (Container Insights) | [dashboard under live traffic](img/dashboard-load.png) · [alarm + recovery email pair](img/alarm-emails.png) |

## Advanced features (assignment: "choose at least one" — three of the four are built)

| Feature | Status | How | Where | Evidence |
|---|---|---|---|---|
| **Auto-scaling policies based on application metrics** | ✅ Built | Two target-tracking policies (CPU @ 60, ALB requests-per-target @ 100); `lifecycle ignore_changes` cedes `desired_count` to the scaler so Terraform and the autoscaler never fight. | `terraform/envs/dev/autoscaling.tf` · `terraform/modules/ecs-service/main.tf` (lifecycle) | [scale-out event](img/scale-2-to-5.png) |
| **Automated security scanning & compliance checks in the pipeline** | ✅ Built — three layers | **tfsec** on both Terraform roots, **Trivy** scans every image before it may enter ECR — the frozen 2016-18 upstream images scan as a documented risk manifest (`.trivyignore`), while a dedicated gate job proves CRITICAL-blocking for anything new (**scan gate demo job under the mirror stage**); **ECR scan-on-push** re-scans at rest. | `.gitlab-ci.yml` (tfsec, mirror, `scan-gate-demo`) · `.trivyignore` · `bootstrap/` (scan-on-push) | [Trivy manifest output](img/trivy-manifest.png) · [blocking gate failing red](img/scan-gate-red.png) |
| **Infrastructure drift detection and remediation** | ✅ Built | pushes to main and scheduled pipeline run `terraform plan -detailed-exitcode`; exit code 2 gives "DRIFT DETECTED" and the offending diff above it. Remediation is deliberately human: revert the out-of-band change, or codify it through the same reviewed, gated apply as everything else. Legitimate drift (autoscaler-owned `desired_count`) is excluded by ownership, so the drift detector and the auto-scaler never fight. | `.gitlab-ci.yml` (`drift-check`) | [drift caught red after a console edit](img/drift-red.png) |
| **Blue-green / canary deployment strategies** | ◐ Objective covered, pattern documented | Not implemented as distinct environments. The *goal* — seamless updates with safe rollback — is delivered by ECS rolling deployments plus the **deployment circuit breaker's automatic rollback**, demonstrated in testing. Full blue-green via CodeDeploy (two target groups, listener flip) is specified as the production path in [05](05-production-roadmap.md). | `terraform/modules/ecs-service` (circuit breaker) | — |

**Beyond the ask:** two additional operational buttons (`load-demo`/`load-stop` for on-demand load with editable parameters, `seed` for one-press database seeding with log-verified success), a double-locked `tf-destroy` (manual **plus** typed confirmation), and OIDC-federated pipeline identity holding **zero stored cloud credentials**.
